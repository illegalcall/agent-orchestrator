1. Problem Statement

⚠️ The Problem

Agent Orchestrator exhausts GitHub's 5,000 requests/hour API limit during heavy parallel usage. With 10+ agent sessions running concurrently, the orchestrator's polling loop consumes 2,000–4,000 API calls/hour — leaving little headroom and frequently hitting rate limits, which kills all sessions.

What Happened (March 22, 2026)

  • 12 agent sessions spawned on issue #13, all sharing the same branch/PR
  • Each session + orchestrator polling burned through API quota
  • Rate limit hit → GitHub returned 403s → orchestrator starved → all sessions killed
  • Result: 25 dead sessions, zero shipped PRs

2. System Architecture — Where API Calls Happen

┌─────────────────────────────────────────────────────────┐ │ ORCHESTRATOR │ │ (single process — polls ALL sessions periodically) │ └────────────────────┬────────────────────────────────────┘ │ Polling loop (every 30s) For EACH active session: │ ┌─────────────┼─────────────┬─────────────┐ ▼ ▼ ▼ ▼ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ │ Agent 1 │ │ Agent 2 │ │ Agent 3 │ │ Agent N │ │ PR #18 │ │ PR #19 │ │ PR #20 │ │ PR #N │ └─────────┘ └─────────┘ └─────────┘ └─────────┘ Each arrow = multiple GitHub API calls: • gh pr view (summary) ─── 1 call • gh pr view (reviews) ─── 1 call • gh pr view (review decision) ─ 1 call • gh pr view (mergeable) ─── 1 call • gh pr checks (CI status) ─── 1 call • gh pr view (threads) ─── 1 call ────────────────────────────────────── Total: ~6 calls per PR per poll cycle

Key Insight

The orchestrator is the bottleneck, not individual agents. It polls GitHub for every active session's PR status in a loop. N sessions × 6 calls × polls/hour = rate limit explosion. Individual agents also make calls (creating PRs, pushing, checking CI), but the orchestrator's continuous polling is the dominant consumer.

4. Phase 1: Request Deduplication & Batching Merged

PR: #617 — fix: reduce GitHub API calls via deduplication and batching

What It Does

  • Request Deduplication: When multiple callers request the same data simultaneously, they share one API call via RequestDeduplicator
  • Field Batching: Combines multiple gh pr view calls (summary, reviews, reviewDecision, mergeable) into a single call via getPRViewData()
  • Duplicate Call Fix: getCISummary() now accepts pre-fetched checks instead of re-fetching

How Deduplication Works

// BEFORE: 5 agents poll same PR simultaneously Agent 1 ──→ gh pr view #18 ──→ GitHub API (call 1) Agent 2 ──→ gh pr view #18 ──→ GitHub API (call 2) Agent 3 ──→ gh pr view #18 ──→ GitHub API (call 3) Agent 4 ──→ gh pr view #18 ──→ GitHub API (call 4) Agent 5 ──→ gh pr view #18 ──→ GitHub API (call 5) // AFTER: RequestDeduplicator collapses identical concurrent requests Agent 1 ──→ dedupeGh("pr view #18") ──→ GitHub API (1 call) Agent 2 ──→ dedupeGh("pr view #18") ──┘ (shares result) Agent 3 ──→ dedupeGh("pr view #18") ──┘ (shares result) Agent 4 ──→ dedupeGh("pr view #18") ──┘ (shares result) Agent 5 ──→ dedupeGh("pr view #18") ──┘ (shares result)

How Field Batching Works

// BEFORE: Separate calls for each field set getPRSummary() → gh pr view --json title,state,body (call 1) getReviewDecision()→ gh pr view --json reviewDecision (call 2) getReviews() → gh pr view --json reviews (call 3) getMergeability() → gh pr view --json mergeable (call 4) getCIChecks() → gh pr checks (call 5) getCISummary() → gh pr checks (duplicate!) (call 6) // AFTER: One batched call + shared CI data getPRViewData() → gh pr view --json title,state,body, reviewDecision,reviews,mergeable (call 1) getCIChecks() → gh pr checks (call 2) getCISummary(checks) → reuses checks from above (call 0)

Limitation

Phase 1 halves the calls per PR, but each PR is still polled independently. With 10+ sessions on different PRs, we still approach the rate limit. Deduplication primarily helps the edge case of multiple sessions sharing the same PR (retry storms).

5. Phase 2: Batched GraphQL Queries Proposed

Core Idea

Instead of making separate REST/CLI calls for each PR, combine all active PRs into a single GraphQL query using field aliases. One request returns data for every PR the orchestrator is tracking.

Architecture Change

// BEFORE: N separate requests (1 per PR) Orchestrator ──→ gh pr view #18 ──→ GitHub REST (call 1) ──→ gh pr checks #18 ─→ GitHub REST (call 2) ──→ gh pr view #19 ──→ GitHub REST (call 3) ──→ gh pr checks #19 ─→ GitHub REST (call 4) ──→ gh pr view #20 ──→ GitHub REST (call 5) ──→ gh pr checks #20 ─→ GitHub REST (call 6) ... × N PRs × 2-3 calls each // AFTER: 1 GraphQL query for ALL PRs Orchestrator ──→ GraphQL batch query ──→ GitHub GraphQL (1 call) ├── PR #18: title, state, CI, reviews, mergeable ├── PR #19: title, state, CI, reviews, mergeable ├── PR #20: title, state, CI, reviews, mergeable └── PR #N: title, state, CI, reviews, mergeable

GraphQL Query Structure

query BatchPRStatus {

      ... same fields
    }
  }

}

Implementation: batchFetchPRStatuses()

async function batchFetchPRStatuses(
  prs: { owner: string; repo: string; number: number }[]
): Promise<Map<number, PRStatus>> {

  // Build aliased GraphQL query
  const fragments = prs.map(pr =>
    `pr${pr.number}: repository(owner: "${pr.owner}", name: "${pr.repo}") {
      pullRequest(number: ${pr.number}) {
        title state mergeable reviewDecision
        reviews(last: 10) { nodes { author { login } state body } }
        commits(last: 1) { nodes { commit { statusCheckRollup {
          state contexts(first: 20) { nodes { ... on CheckRun { name conclusion status } } }
        } } } }
        reviewThreads(first: 20) { nodes { isResolved } }
      }
    }`
  );

  const query = `query BatchPRStatus { ${fragments.join("\n")} }`;

  // Single API call for ALL PRs
  const result = await gh(`api graphql -f query='${query}'`);

  // Parse and return mapped results
  return parseBatchResponse(result, prs);
}

GraphQL Complexity Budget

GitHub GraphQL has a ~500,000 point complexity limit per query. Each PR with full status checks costs approximately 50–150 points. This means we can safely batch 50+ PRs in a single query. For extreme scale (100+ PRs), we'd split into 2–3 queries — still vastly better than 100+ individual calls.

6. Phase 3: Webhook-Driven Updates Future

The Ultimate Solution: Stop Polling

Instead of polling GitHub every 30 seconds, register webhooks so GitHub pushes updates to us. Zero polling = zero wasted API calls. The orchestrator only makes API calls when it needs to take action (merge, comment, etc.).

// CURRENT: Orchestrator polls GitHub continuously Orchestrator ──(every 30s)──→ GitHub: "Any changes?" GitHub: "No." Orchestrator ──(every 30s)──→ GitHub: "Any changes?" GitHub: "No." Orchestrator ──(every 30s)──→ GitHub: "Yes! CI passed." (wasted 100+ calls to find this out) // WEBHOOK: GitHub tells us immediately GitHub ──(CI completed)──→ Orchestrator webhook endpoint Orchestrator: "Got it. Updating session ao-42." (0 polling calls, instant reaction)

Webhook Events Needed

EventReplacesUse Case
check_run / check_suiteCI status pollingKnow instantly when CI passes/fails
pull_request_reviewReview pollingRoute review comments to agents
pull_requestPR state pollingDetect merges, closes, updates
issue_commentComment pollingReact to PR comments

Complexity Trade-off

Webhooks require a publicly accessible endpoint (or a tunnel), webhook secret management, and handling delivery failures. This is the most effective solution but also the most complex to implement. Best suited as a Phase 3 optimization after Phase 2 proves the batching approach.

Summary

The rate limit problem is fundamentally a polling fan-out issue. Phase 1 reduces calls per PR. Phase 2 eliminates the fan-out entirely by batching all PRs into one query. Phase 3 eliminates polling altogether. Each phase is independently valuable and incrementally deployable.

Recommended next step: Implement Phase 2 (batched GraphQL) — it delivers the biggest bang for the buck with moderate implementation effort, taking us from ~3,600 calls/hour to ~120 calls/hour at 10 sessions.
